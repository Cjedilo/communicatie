Security model

A technical overview of how Messages establishes trust, authenticates peers, and protects data in transit.

Transport encryption TOFU cert pinning Certificate rotation Peer authentication User authentication Access control

Transport encryption

All traffic — browser to server, and server to server — runs over TLS. There are no plaintext connections anywhere in the stack. The WebSocket protocol used for federation (wss://) and the browser interface (https://) both require TLS.

TOFU certificate pinning

Trust On First Use (TOFU) is the same model SSH uses for host keys. When two servers connect for the first time, the connecting server records the SHA-256 fingerprint of the remote server's TLS certificate. On every subsequent connection, the fingerprint is checked against the stored value.

If the fingerprint changes unexpectedly — without a valid succession record (see below) — the connection is refused and an error is logged. This prevents a third party from intercepting the connection by presenting a different certificate.

TOFU deliberately trades a first-connection leap of faith for strong protection on all future connections. If you need stronger first-connection guarantees, verify the fingerprint out-of-band with the peer's admin.

Certificate rotation

When a server renews its certificate — whether self-signed via the UI or via Let's Encrypt — it creates a succession record: a signed statement using the old private key that vouches for the new certificate's fingerprint.

The record is published at /succession. When a peer reconnects and finds a changed fingerprint, it fetches the succession chain and verifies each step cryptographically. If the chain is valid, the new fingerprint is accepted and stored. If not, the connection is refused.

This allows certificate rotation without manual intervention on peer servers, even if the peer was offline during the rotation.

Peer authentication

After the initial handshake, each server issues the connecting peer a shared secret token. This token must accompany every subsequent federation request. Tokens are stored in the database and checked with constant-time comparison to prevent timing attacks.

This means a server that knows another server's address cannot impersonate it — it also needs the token that was issued during the approved handshake. Spoofing the from field in a message is rejected at every privileged endpoint.

User authentication

Passwords are hashed with Argon2id, the winner of the Password Hashing Competition and the current recommended standard. Parameters are automatically upgraded if the hashing cost settings improve in a future release — existing passwords are transparently rehashed on next login.

Sessions use a 48-byte cryptographically random token stored in a Secure; HttpOnly; SameSite=Strict cookie. Session duration is configurable; expired sessions are purged periodically.

Login and registration endpoints are rate-limited per IP. When running behind a trusted reverse proxy, the real client IP is read from X-Forwarded-For — only from proxy IPs explicitly listed in Advanced → Trusted proxies.

Access control

Access checks are enforced at every layer:

  • Private channels — federation endpoints verify that the requesting peer has at least one member in the channel, and individual message fetches additionally verify per-user membership.
  • Message content — the get_message endpoint only serves content from channels the requesting peer can access. It never serves content received from another peer (only content this server originated).
  • Image proxy — the /proxy_img endpoint only fetches from known peer addresses, preventing server-side request forgery against internal services.
  • Owner-only settings — server configuration endpoints verify the session belongs to the designated owner account before accepting changes.
