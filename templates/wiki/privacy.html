Privacy & data

What Messages stores, where it stores it, and what connected servers can and can't see.

Your server What peers see Content ownership Images Metadata

Your server

Your server stores:

  • User accounts (username, hashed password, display name, avatar)
  • Channel definitions and membership lists
  • Messages sent by users on this server — text and images
  • Message index entries for messages received from peers (metadata only — no content)
  • Peer connection records (address, name, certificate fingerprint, auth token)
  • User settings (stream preferences, blocked users)

Nothing else. There is no analytics, no telemetry, no external reporting of any kind. The software does not phone home.

What peers see

A connected peer server can:

  • Read and post in your public channels
  • Read and post in private channels where one of their users has been explicitly added as a member
  • See your public user list (names and avatars)
  • Fetch content for messages they have legitimate access to

A connected peer cannot:

  • Read private channels they have no members in
  • Read direct messages between users on your server
  • Access any content without presenting a valid auth token issued during the approved handshake

Content ownership

When someone on a peer reads a message you sent, their server fetches the content from yours at that moment. Nothing is cached on their side.

This means you retain control. If you block a peer, they immediately lose access to your content — both future messages and on-demand fetches of old ones. If your server goes offline, your messages become temporarily unavailable to peers. This is intentional: the writer owns the content.

It also means deletion is real. Remove a message or an image from your server and it's gone — not just hidden, not archived somewhere else. Anyone who already has it on screen may still see it until they navigate away, but the next time anyone tries to fetch it, it simply isn't there.

Images

Images you upload are stored on your server's filesystem, in the directory configured under Server → Image storage. Peers fetch images through a proxy on their own server — your images are never stored on their disk.

If you delete an image file from the filesystem, it becomes unavailable everywhere — including to users on your own server.

Metadata

One nuance: when a message is sent to a channel hosted on another server, that server stores a message index entry — enough to know a message exists (sender, timestamp, channel) without the content itself. This metadata persists on the host server even if the originating server goes offline or the peer relationship ends.

This is necessary for the host server to maintain a coherent message timeline. It mirrors how email works — the receiving server records that a message arrived, even if the original sender's server later disappears.
