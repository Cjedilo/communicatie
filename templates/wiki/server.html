Server management

Everything in ⚙️ Server is only visible to the server owner. This is where you configure how your server presents itself, stores data, and handles connections.

Identity

Your server has two identifying properties:

  • Name — how your server appears to peers and in the users list. Pick something recognisable.
  • Address — the full wss:// address other servers use to connect to you, e.g. wss://myhome.example.com/chat. This is what you share with other server admins when setting up federation. Regular users use the https:// version of the same address.

Changing the address affects federation — existing peers will need to reconnect at the new address. The base path (e.g. /chat) is derived automatically from whatever path you put in the address.

Image storage

Uploaded images — avatars and photos sent in channels — are stored in a directory on the server's filesystem. The current directory and available disk space are shown here.

You can change the directory at any time. When you do, you'll be offered three options:

  • Keep files in place — the old files stay where they are; new uploads go to the new path. Old images may become unreachable.
  • Copy to new location — all existing files are copied. Both locations keep their files.
  • Move to new location — all existing files are moved. Frees up space at the old path.

Database

Messages supports two database backends:

  • SQLite — stores everything in a single file. Simple to set up, easy to back up, fine for most deployments.
  • PostgreSQL — better for larger installations with many users or high message volume. Requires a running PostgreSQL server.

Data is not migrated automatically when switching backends. Back up first, then switch. A restart is required to apply the change.

SSL certificate

All connections to your server are encrypted. By default, Messages generates a self-signed certificate — this works immediately but causes a browser warning on first visit. See Getting started for why that warning isn't as scary as it looks.

There are two options:

  • Self-signed, auto-generated — the default. Use Renew to generate a fresh certificate. Existing peers are notified automatically and stay connected via the succession mechanism.
  • Your own certificate — enter paths to your .pem files (e.g. from Let's Encrypt or certbot). Use Reload after the tool has renewed the files on disk — no restart needed.

The expiry date is shown next to the certificate info. A warning badge (!) appears on the Server icon when renewal is overdue.

Peer policy

The peer policy controls which other servers can connect to yours. Set it under 🔗 Peers:

  • Open — any server can connect automatically
  • Approval required — you review and approve each new server before it can connect
  • Closed — no new connections are accepted

Existing approved peers are not affected when you change the policy — only new connection attempts are evaluated against the current setting.

Connecting to a peer

Go to 🔗 Peers and paste the wss:// address of another Messages server. Your server connects to theirs, exchanges names and establishes trust, and the connection is live.

The other server's public channels then appear under Remote channels in the Channels page, and their users appear in the Users page. You can also invite each other's users to private channels.

On first connect, both servers pin each other's certificate fingerprint. If the fingerprint ever changes without a valid succession record, the connection is refused. See Security model for details.

Advanced settings

The Advanced link at the top of the Server page gives access to lower-level settings:

  • Network — bind host and ports (requires restart)
  • Rate limiting — max login attempts and messages per minute, trusted proxy IPs (applied immediately)
  • Federation — peer connection timeout (applied immediately)
  • Sessions — how long login sessions stay valid (requires restart)
  • Uploads — maximum image upload size (requires restart)
