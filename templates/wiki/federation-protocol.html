Federation protocol

How two Messages servers communicate — the wire protocol, message flow, and design decisions behind federation.

Transport Handshake Message flow Content ownership Subscriptions Persistent sessions

Transport

Federation uses WebSocket over TLS (wss://). Each server exposes a /peer endpoint. All messages are JSON objects with a type field. There is no binary framing or custom protocol — a wscat session is sufficient to inspect the wire.

Handshake

When server A connects to server B for the first time:

  1. A opens a WebSocket to B/peer and sends a peer_details message with its own address and name.
  2. B checks its peer policy (open / approval / closed) and responds.
  3. If approved, B issues A a shared secret token in the response. A stores this token and attaches it to every subsequent request.
  4. A records B's TLS certificate fingerprint (TOFU). Future connections verify against this fingerprint.

On reconnection the handshake is skipped — the persistent session reuses the stored token.

Message flow

When a user on server A posts to a channel hosted on server B:

  1. A sends a new_message request to B with the message metadata, sender identity, and text content.
  2. B stores the message index (who, when, which channel) and pushes a notification to any local clients watching that channel.
  3. B does not store the message content — only A holds the content. B stores only enough to know a message exists.

When a user on B later reads that message, B fetches the content from A on demand via get_message. If A is offline, the message shows as temporarily unavailable.

Content ownership

This is the central design decision of the protocol: the originating server is the sole source of truth for message content.

Remote servers store message metadata (index entries) but never cache content. This means:

  • The author controls their content — going offline makes old messages temporarily unavailable; blocking a peer makes them permanently unavailable to that peer.
  • There is no content synchronisation problem — there is only one copy.
  • Storage requirements on receiving servers are minimal.

Images work the same way. The 📷 sentinel in the message index tells receiving servers an image exists; the actual file is fetched via the image proxy when displayed.

Subscriptions

Servers can subscribe to channels or to the full stream of a peer:

  • subscribe — receive live new_message pushes for a specific channel.
  • subscribe_stream — receive all new messages across all accessible channels on that peer.

Both require a valid auth token. Private channel subscriptions additionally require that the subscribing peer has at least one member in that channel.

Persistent sessions

Outbound connections are pooled — one persistent WebSocket per peer, reused across requests. This avoids the overhead of a TLS handshake per request and allows pipelined operations (multiple requests in flight on one connection under a lock).

Sessions idle for more than 10 minutes are closed automatically. If a session fails mid-request (timeout, network error), the socket is dropped immediately so the next request reconnects cleanly rather than reading a stale response.

Tokens are attached automatically to every outbound request by the session pool, so individual call sites don't need to handle authentication.
